## Full Content

- Immediately create documentation when user requests saving work (any trigger phrase)
- Proactively offer documentation at natural completion points (working systems, configured automation)
- Use standard offer: 'This is complete and working. Should I create documentation before we move on?'
- Document in appropriate location: /docs/ for systems, /clients/[client]/CONTEXT.md for client insights, /tools/ for architecture
- Include all required sections for system documentation: Overview, How It Works, Files, Running Manually, Monitoring, Troubleshooting, Test Results, Implementation History
- Update client CONTEXT.md by adding only, never removing; include dates and source references
- Create summary files for major projects showing all components, file locations, and complete command reference
- Capture client context items: strategic decisions, KPIs, preferences, business changes, performance patterns, experiments, dates, concerns, competitive intelligence
- Document tool architecture decisions, dependencies, API integrations, data flows, configuration, limitations, and future enhancements
- Use clear file naming: SYSTEM-NAME-COMPLETE.md for finished implementations, SYSTEM-NAME.md for reference, TOOL-NAME-GUIDE.md for user guides
